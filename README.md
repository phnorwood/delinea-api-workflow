# delinea-api-workflow

Minimal EC2 + NGINX deployment using **Terraform** (provisioning) and **Ansible**
(configuration), orchestrated with the **AWS CLI**. AWS credentials are retrieved
at apply time from **Delinea Secret Server** — they are never stored in
`terraform.tfvars` or in Terraform state.

## What it deploys

- One **t3.micro** EC2 instance (smallest sensible default; override via `instance_type`)
- Amazon Linux 2023, in the account's default VPC (no extra networking)
- A security group allowing SSH (22) and HTTP (80)
- An auto-generated SSH keypair for Ansible
- **NGINX** installed, enabled, and serving a landing page

## Layout

```
terraform/   provisions the instance, SG, keypair; renders the Ansible inventory
ansible/     playbook that installs and starts NGINX
```

## AWS credentials via Delinea Secret Server (TSS)

AWS credentials live in a **Delinea Secret Server** secret and are fetched by
Terraform at apply time using the [`DelineaXPM/tss`](https://registry.terraform.io/providers/DelineaXPM/tss/latest/docs)
provider. Nothing sensitive is written to `terraform.tfvars` or to
`terraform.tfstate`.

How it works (see [terraform/main.tf](terraform/main.tf)):

- The `tss` provider authenticates to Secret Server with an OAuth **bearer
  token**, supplied out of band via the `TF_VAR_tss_token` environment variable.
- Two **ephemeral** `tss_secret` resources read the access-key and secret-key
  fields from the secret. Ephemeral values exist only in memory during the run —
  they are never persisted to state or disk (requires Terraform >= 1.10).
- The `aws` provider is wired to those ephemeral values instead of to static
  variables.

### Configure

Non-secret settings go in `terraform.tfvars`; the token is passed via the
environment:

```bash
cp terraform/terraform.tfvars.example terraform/terraform.tfvars
# edit terraform/terraform.tfvars — set tss_server_url and tss_secret_id
export TF_VAR_tss_token="<your Secret Server bearer token>"
```

`terraform.tfvars` and the generated `*.pem` key are gitignored.

### TSS variables

| Variable | Where set | Description |
|---|---|---|
| `tss_server_url` | tfvars | Secret Server base URL. On-prem: `https://host/SecretServer`; Cloud: `https://<tenant>.secretservercloud.com` (no path). |
| `tss_token` | `TF_VAR_tss_token` env | OAuth bearer token, provided out of band. Sensitive. |
| `tss_secret_id` | tfvars | ID of the secret holding the AWS credentials. |
| `aws_access_key_slug` | tfvars (optional) | Field slug for the access key. Default `access-key`. |
| `aws_secret_key_slug` | tfvars (optional) | Field slug for the secret key. Default `secret-key`. |

The field slugs come from the secret's template. If your secret uses different
slugs (e.g. `username` / `password`), override them. To check the slugs on a
secret:

```bash
curl -sS -H "Authorization: Bearer $TF_VAR_tss_token" \
  "$TSS_SERVER_URL/api/v1/secrets/<id>" | jq '.items[].slug'
```

## Deploy

```bash
export TF_VAR_tss_token="<your Secret Server bearer token>"
cd terraform
terraform init                      # installs the tss provider
terraform apply                     # fetches AWS creds from Secret Server, then provisions
cd ../ansible
ansible-playbook -i inventory.ini playbook.yml   # inventory is generated by terraform
```

## Tear down

```bash
cd terraform && terraform destroy
```

## Troubleshooting (TSS)

- **`The terraform-provider-tss plugin crashed!` / nil pointer in `accessResource`.**
  Almost always an unreachable or wrong `tss_server_url` (e.g. a placeholder host
  that doesn't resolve). The tss SDK doesn't check the transport error before
  reading the HTTP status, so a failed request surfaces as a plugin crash rather
  than a clean message. Verify the host answers first:
  ```bash
  curl -sS "$TSS_SERVER_URL/api/v1/healthcheck"     # expect HTTP 200
  ```
- **`Secret Fetch Error … 401 Unauthorized`.** The bearer token is missing,
  expired, or lacks View access to the secret. Refresh `TF_VAR_tss_token`.
- **Empty value or fetch error on a field.** The `aws_access_key_slug` /
  `aws_secret_key_slug` don't match the secret's template. Inspect the real slugs
  with the `curl … | jq '.items[].slug'` command above.
